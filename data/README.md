# Data layout (populate & ingest)

Use this split so **demo-safe** stuff is commit-friendly and **real exports** never hit Git.

## 1. Structured fixtures → direct DB rows

| Location | Purpose |
|----------|---------|
| **`demo/sample_data.json`** | Canonical JSON array matching `knowledge_items` columns; **`npm run demo:load`** reads this. |
| **`data/fixtures/*.json`** | Optional extra seed files. Add a small loader script or merge into `demo:load` when needed. |

Shape matches **`docs/PROGRESS.md`** API contract (category, `action_items`, `source_type`, etc.).

## 2a. 小红书-style CN posts → structured JSON (rule-based)

| Location | Purpose |
|----------|---------|
| **`data/raw_posts/*.txt`** | **Put demo 小红书 copies here** — one post per `.txt`. See **`data/raw_posts/README.txt`**. |
| **`data/local/raw_posts/`** (optional) | Same as above but **private** — folder is gitignored; run parser with `-i path`. |
| **`data/output/seed_posts.json`** | Generated by the parser (**gitignored** — run locally). |

```bash
# From repo root (Python 3, stdlib only)
python3 scripts/ingest/parse_cn_us_posts.py
# or: npm run ingest:parse-cn

python3 scripts/ingest/parse_cn_us_posts.py --input data/raw_posts --output data/output/seed_posts.json
```

Each JSON object includes: `title`, `content`, `city_cn`, `state_cn`, `store_name`, `category`, `price_level`, `vibes`, `tags`, `signals` (plus `source`, `language`, `region`, `filename`).

**Populate Supabase (after `seed_posts.json` exists):**

```bash
# Direct insert — maps parser → knowledge_items (no MiniMax; category 咖啡店→Food, etc.)
npm run ingest:seed -- --mode db

# Via API — start `npm run dev` first; MiniMax re-extracts each row (cost + latency)
npm run ingest:seed -- --mode api --delay-ms 1200
```

- **Unified schema (later)** — one shape for TikTok / WeChat / chat: `source`, `type`, `raw_text`, `tags`, `entities`, `signals`, `timestamp` (see **GOAL** honest ingestion).

## 2b. WeFlow (optional) — chat export → this repo

[WeFlow](https://github.com/hicccc77/WeFlow) is a **local** app to view / analyze / **export** WeChat chats (TXT, JSON, HTML, etc.). Releases: [github.com/hicccc77/WeFlow/releases](https://github.com/hicccc77/WeFlow/releases). Docs / notes: [doc.weflow.top](https://doc.weflow.top/).

**Requirements (from upstream):** WeChat **4.0+**; macOS build is **Apple Silicon (arm64)** `.dmg` — check compatibility with your Mac.

**Important — your `Backup/.../files` folder:** That tree is **WeChat’s own backup package** (opaque chunks + indexes). WeFlow is built around **live** WeChat data and a **database connection inside the app** — it is **not** documented as “select this Backup folder and read it like a folder of `.txt`.” In practice you usually:

1. **WeChat 4.0 installed and logged in** on the same Mac, and  
2. **WeFlow** connected to that data (first-run / settings — follow WeFlow’s UI), **then**  
3. **Export** the chats you want as **`.txt` or `.json`**.

If you only have a **cold backup** and WeFlow does not see history, you may need to **restore that backup with WeChat’s own restore flow** first (so the data lives in WeChat’s active store), then open WeFlow again. If WeFlow still shows nothing, treat **manual copy‑paste** into `data/raw_posts/*.txt` as the fallback.

**After export:**

```bash
# Save exports somewhere under data/local/ (gitignored), e.g. data/local/weflow/*.txt
# API running: npm run dev
USER_ID=<uuid> npm run ingest:local -- --delay-ms 800 "/absolute/path/to/weflow-exports"
```

Optional: WeFlow **HTTP API** on `http://127.0.0.1:5031` (Settings → API) — see [HTTP-API.md](https://github.com/hicccc77/WeFlow/blob/main/docs/HTTP-API.md); use **`Authorization: Bearer <token>`** on `/api/v1/sessions` and `/api/v1/messages`. You can script pulls into a `.txt` file, then use **`ingest:local`** as above.

**Do not** commit raw chat exports to Git.

## 2. Raw text samples (small, OK to commit)

**`data/samples/`** — short `.txt` / `.md` files for:

- Pitch demos (`npm run ingest:local -- data/samples`)
- Showing teammates what a “good paste” looks like

Keep files **tiny** and **non-private** (no real phone numbers, addresses, or DMs).

## 3. Local-only (never commit)

**`data/local/`** — put here:

- WeChat backup exports you copied out as `.txt`
- TikTok / 小红书 personal-data ZIPs **after** you unzip (or point `ingest:local` at the unzip folder)
- Any “scraper” **output** you generated on your own machine

Everything under `data/local/` is **gitignored**. Share paths only in chat with consent, not via the repo.

## 4. UI demo strings (not files)

**`web/src/data/demoIngestSamples.ts`** — multiline strings for **`/connect`** “load sample” behavior. Same pipeline as paste, but lives in code for one-click demos.

## 5. Scripts (no live “scrapers” in-repo)

| Script | Role |
|--------|------|
| **`scripts/bulk-ingest-local.ts`** | Walks folders of `.txt` / `.md` / `.log` and POSTs chunks to **`POST /api/message`** (API must be running). |
| **`scripts/ingest/parse_cn_us_posts.py`** | **`data/raw_posts/`** → **`data/output/seed_posts.json`** (rule-based CN + US city extraction). |

**Convention:** Do **not** add scripts that hammer 小红书 / TikTok / WeChat **live HTML** in CI or as a default hackathon path. Prefer:

- User **export** or **paste**
- **Photon** for iMessage (macOS)
- Optional **browser extension** later (user-initiated capture)

If you write a one-off parser for a **downloaded** file format, put it under **`scripts/ingest/`** and document the **exact** filename pattern it expects.

## Quick commands

```bash
# API up first: npm run dev

# Ingest committed samples (dry run first)
USER_ID=<uuid> npm run ingest:local -- --dry-run data/samples
USER_ID=<uuid> npm run ingest:local -- data/samples

# Your private exports (path on disk, not in git)
USER_ID=<uuid> npm run ingest:local -- "$HOME/.../wechat-export-folder"

# Structured rows → Supabase (no MiniMax per row — already structured)
USER_ID=<uuid> npm run demo:load
```
